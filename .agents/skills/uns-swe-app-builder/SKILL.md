# UNS-SWE App Builder

## Purpose

Generate complete, runnable, production-quality applications from requirements documents. Prefer `specs/spec.md` from `$industrial-requirements-srs` as the clearest build input when available. Build on existing project files when present, choose implementation patterns that match the detected stack, and verify the app can run before handing it back.

## Workflow

1. Inspect the working directory before creating files.
2. Identify whether an existing template or app is present and detect its stack.
3. Check whether `specs/spec.md` exists before planning implementation.
4. If `specs/spec.md` exists, read it before any raw PRD, notes, or informal requirements and treat it as the primary implementation source.
5. If no `specs/spec.md` exists, read the requirements document or user-provided spec carefully.
6. Read `references/implementation-checklist.md` before planning substantial implementation work.
7. Plan the file structure and implementation order.
8. Implement the full app, wiring frontend, backend, data models, validation, and configuration as required by the spec. When building frontend UI in this scaffold, use `$uns-swe-ui-generation` for scaffold-native Tailwind/Tier0 patterns and `$app-i18n-copy` for product-copy consistency.
9. Self-review for syntax, imports, placeholders, TODOs, missing flows, security, and error handling.
10. Run the relevant install/build/test/dev-server workflow available in the environment.
11. If preview MCP tools are available and a meaningful code change is complete, use the preview workflow before telling the user the preview is ready.

## Existing Files First

- Always inspect the output or working directory before creating new files.
- If a project already exists, build on top of it instead of starting from scratch.
- Preserve the existing technology stack, conventions, routing, component style, file naming, and package manager unless the requirements explicitly demand a change.
- Do not overwrite unrelated user changes.
- If the directory is empty, create a conventional project structure appropriate to the requirements and selected stack.

## Requirements Handling

- Prefer `specs/spec.md` generated by `$industrial-requirements-srs` because it should already contain implementation-relevant roles, stories, data models, business logic, UI flows, and validation rules.
- If both `specs/spec.md` and PRD/raw notes are present, treat `specs/spec.md` as the primary source and use other materials only to resolve context that does not conflict with the SRS.
- Do not prefer a raw PRD, app prompt, or user notes over `specs/spec.md`.
- If no SRS file is present, continue from the provided PRD, requirements document, or structured prompt; do not block solely because the preferred SRS handoff is missing.
- If requirements are too vague to build safely, suggest first using `$industrial-requirements-srs` to clarify and structure them.
- Treat the provided requirements document, SRS, PRD, or structured prompt as the source of truth.
- Implement all required features, roles, workflows, validation rules, UI flows, and data interactions that are relevant to the app.
- If the requirements specify an output directory or session ID, organize generated code there.
- If essential implementation details are missing but can be reasonably inferred from the existing stack and spec, proceed conservatively.
- Ask only when a missing choice would materially change the app architecture, core workflow, or user-facing behavior.
- Do not leave placeholder features, TODO-only stubs, fake buttons, or disconnected views for required functionality.

## Implementation Rules

- Start with foundational files such as package/config files, models, schemas, shared types, utilities, and route structure.
- Build UI screens and components around the required user workflows, not around decorative landing-page sections.
- Implement backend/API logic, persistence adapters, validation, and error handling when the requirements call for them.
- In this TanStack Start MES scaffold, rely on the shared service-layer
  `bootstrapModule(...)` helper for runtime schema/table/index creation and
  baseline seed. Do not write custom create/seed ordering logic in each
  generated service.
- When generating runtime baseline seed for this scaffold, build a valid FK
  graph explicitly before calling `insert().values([...])`: declare parent rows
  and IDs first, insert parent tables before child tables, and set every
  non-null child FK from a named parent ID or `requireSeedRef()` /
  `requireSeedValue()` from `@/services/seed-utils`. Never omit required FK
  properties and never allow `undefined` in seed values; Drizzle emits SQL
  `default` for missing/undefined object properties, which breaks child inserts
  such as `sales_order_items.sales_order_id`.
- In generated Drizzle services for this scaffold, avoid relational helpers
  such as `db.query.<table>.findFirst()` and `tx.query.<table>.findFirst()`.
  Use explicit builder queries for single-row lookups:
  `select().from(table).where(...).limit(1)`, then read `rows[0]`. Use
  `tx.select()` inside transactions.
- For generated frontend request hooks or page-local loaders, key effects by
  stable primitive request keys instead of inline loader function identities.
  A reusable hook should look like `useRequest(requestKey, loader)` so a
  successful `setState` does not recreate `() => fetchJson(...)` and trigger an
  identical fetch loop.
- Connect files through real imports, routes, handlers, and state flows.
- Include clear comments only where they explain non-obvious behavior.
- Avoid common security issues such as unsanitized HTML injection, unsafe auth assumptions, leaking secrets, and injection-prone data handling.
- Keep code syntactically correct, idiomatic for the stack, and production-oriented.

## Preview Workflow

If `preview_start`, `preview_logs`, or `preview_restart` MCP tools are available, use them for preview instead of manually managing the dev server.

- Do not run `npm install` manually when managed preview install is available; it can race with the managed install and corrupt `node_modules` on the shared volume.
- After meaningful code changes, call `preview_start`.
- If `preview_start` returns `state: "failed"`, call `preview_logs`, fix the issue, then call `preview_restart`.
- After installing a new npm dependency through an approved workflow, call `preview_restart`.
- Tell the user `preview is ready` when the managed preview is ready.
- Do not paste the returned loopback URL to the user; it is for internal self-check only and the UI delivers the public preview separately.

If preview MCP tools are unavailable, use the repository's normal local verification workflow and report what ran.

## Completion Criteria

Before finishing:

- Verify all required functionality is implemented.
- Verify generated files are connected through imports, routes, handlers, and configuration.
- Remove placeholder code, unused TODOs, and dead scaffolding.
- Run available checks or explain why they could not be run.
- Summarize the implemented app and verification result concisely.
