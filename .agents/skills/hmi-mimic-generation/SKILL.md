# HMI Mimic 生成（需求 → 工艺流程图 schema）

## 定位

本模版的整个 SCADA 画面由**一份 mimic JSON** 驱动（nodes/edges/interlocks）。生成客户应用 = 产出这份 JSON 并持久化。规范文档是权威，本 skill 只给工作流、决策规则和文档没写的坑：

- `docs/hmi-schema.md` — 字段级规范（节点/连线/联锁/绑定 key 语义/最小示例）。注意其字段表不全：节点另有 `watches`/`actions`/`publishPresets`，连线另有 `lead`/`fromSide`/`toSide` 等已实装字段——**zod `src/hmi/schema/schema.ts` 才是完整真相源**
- `docs/ai-image-to-schema.md` — P&ID 图片 → 拓扑的映射规则
- `src/hmi/symbols/capabilities.ts` — **图元类型与可绑状态的唯一真相源**。文档里的类型表会滞后（曾出现文档 24 类 vs 实际 27 类），选型前必读这份代码，不要只信文档表格
- 参考实物：`src/hmi/data/demo-mimic.json`（T-104 乙二醇精馏演示样板）——布局间距、位号风格、连线/选型惯例都以它为基准（**只读参考，绝不在其上改、也不 copy 来当客户图**）。⚠️ `default-mimic.json` 现为**空白图**（模板开局空画布、DB 空表种子），不是生成目标

## 核心原则：只产拓扑，不产状态

需求/图纸给的是静态拓扑（什么设备、怎么连）。**不要猜运行态**（开/关、运行/停止）——运行态来自 MQTT，属于数据绑定步骤（$hmi-data-binding）。没有绑定的设备渲染为去能量化基线态 + 「未配置」虚化提示，这是预期行为不是 bug。

## 拓扑产出后自动接绑定（落完拓扑直接读 UNS 配 topic，别停下来等用户发话）

🔁 **「图/需求 → 拓扑」只是上半程；交付的图必须能接真实数据。** 拓扑落完后**主动衔接 $hmi-data-binding**，把读 UNS 树、给元件配 topic 当成同一个任务的下半程**自动做掉**——**不要**把「绑定」当成需用户另行开口的独立请求。用户给了图（尤其含位号/数值的工业图）= **默认要这张图能连真实数据**，无需再说一遍「去配 topic」「读 UNS 树」。漏掉这步 = 交付一张全是「未配置」空壳的图，是本环节最常见的不完整交付。

**判定与动作（拓扑落完立即执行，无需追问用户）：**
1. **先探 UNS 是否可用**：调 `browseUnsFn({ path: "", maxDepth: 1 })`（或 `searchUnsFn` 任意位号）。返回 `available:false`、或 env 未配（缺 `TIER0_API_HOST`/`TIER0_API_KEY`）→ 停在拓扑，在交付说明写「UNS 未配，topic 待补」。
2. **UNS 可用 → 立即逐台按位号读树配 topic**：`searchUnsFn` 按位号搜、`browseUnsFn` 看命名层级、`readUnsFn` 取当前值定值域。完整决策树 A→B→C→D、绑定结构、回读验真值全在 **$hmi-data-binding**——**直接调用那个 skill 往下做，别让用户先开口**。
3. **确有匹配不到的设备** → 该字段 `bindings` 留空（「未配置」虚化）+ 列入交付说明待补清单；**绝不假绑占位 topic**（占位 topic 无数据 → 永远 stale / 满屏同值）。

> **唯二停在拓扑的情况**：① 用户明确说「只要拓扑 / 先别绑数据」；② UNS 不可用（`available:false` 或 env 缺）。其余一律：拓扑落完 → 自动读 UNS → 绑定，一气呵成，不用户催。

## 工作流

1. **盘点设备 → 选型**：逐台对照 `capabilities.ts` 选 `type`。易混项：
   - 立式储罐=`tank`；**卧式缓冲罐/分离罐=`drum`**；反应釜/带搅拌=`vessel`(+`props.agitator:true`)；固体料仓=`silo`
   - 手动阀/开关阀=`valve`；带执行器连续调节=`controlvalve`；止回=`checkvalve`；安全阀=`safetyvalve`
   - 现场变送器（LT/PT/FT/TT）=`instrument`（`props.tag`），不是 `dialgauge`（盘面表才用）；FT/LT 等变送器=`instrument`（ISA 圆圈气泡，画在管道外经 `lead` 引线连设备）；`meter`=管段内置内联表头（无引线直接串在管线上）。有引线测量用 `instrument`，管段内置用 `meter`
   - **工艺边界 I/O（进料/产品/放空/公用工程/采出/REF/RES 等"图边的箭头或框"）=`terminal`**：绑 `flow`，`rotation` 定料流朝向（0=尖朝右、180=朝左、90=朝下…）。别用 `tank` 充当出入口
   - **裸数值标注（直接贴在管线/设备上的温度/压力读数，如 0.8℃/145.7℃）=`readout`**：绑一个 `value`+单位，渲染到最上层盖住组件（overlay 层）。位号留 `label`（检视面板看），画布只显值
   - **🗂 全 27 类组件速查（选型先过一遍全表，别只用熟的几种——只会 exchanger/instrument/readout 是画错+画面单调的头号原因）**：

     | 类别 | type | 何时用（按功能 + 源图画出的形状）|
     |---|---|---|
     | 容器 | `tank` | 立式储罐（锥顶平底，液位填充）|
     | | `vessel` | 立式容器/反应釜（封头曲面+夹套；`props.agitator:true` 加搅拌）|
     | | `drum` | **卧式**缓冲罐/分离罐（底部液位）|
     | | `column` | 精馏/吸收塔（塔盘内件+塔釜液位，瘦高，等比缩放）|
     | | `silo` | 固体料仓（锥斗）；`cyclone` 旋风分离器（短柱+长锥）|
     | 换热 | `exchanger` | **卧式**管壳换热器（TEMA）|
     | | `condenser` | **立式**冷凝器 |
     | | `cooler` | 空冷器（管束+风道，带风机 `running`）|
     | | `heater` | 加热器（加热盘管，`running`+`temp`）|
     | 设备 | `pump`/`motor`/`fan`/`compressor` | 离心泵 / 电机 / 蜗壳风机 / 压缩机（各自机壳形状不同，按源图选）|
     | | `agitator`/`mixer`/`filter` | 顶置搅拌器 / 管段静态混合器 / 过滤器（压差 `dp`）|
     | 执行器 | `valve`/`controlvalve` | 手动开关阀（开关二态）/ 调节阀（膜头，开度 0–100）|
     | | `checkvalve`/`safetyvalve`/`damper`/`switch` | 止回阀（方向箭头）/ 安全阀（角式+弹簧）/ 风门（叶片角）/ 刀闸（开合度）|
     | 仪表 | `instrument` | ISA 仪表，**经引线**连设备：DCS 方框(位号+值,默认) 或 ISA 圆气泡(`props.display:"bubble"`)|
     | | `meter` | 管段**内联**流量计（无引线，直接串在管线上）|
     | | `dialgauge`/`bargauge` | 表盘指针表（盘面字母 `props.face`）/ 竖条填充表 |
     | | `readout` | 裸数值标注（叠在设备/管线上的温度/压力数字，overlay 层）|
     | 端子 | `terminal` | 工艺边界 I/O + 介质/工段文字（进料/产品/放空/公用工程/CWR/至XX），`rotation` 定料流朝向 |

     **选型铁律**：先按「源图画的是什么设备、什么形状」在**全表**找最贴的 type，拿不准查 `capabilities.ts` 的 `desc`/`effect`。**最像的形状优先**——卧式换热器≠立式冷凝器≠空冷器，调节阀≠手阀≠止回阀，内联流量计≠引线变送器；别因为只熟那两三种就把它们全塞成 exchanger/instrument，那正是画面失真又单调的根因。
   - **用足每个组件的显示能力（别只摆造型、留空壳）**：每个 node 渲染时会显示——① `label`（设备名/位号，画在本体下方）② `inline` 列出的绑定字段实时值（泵 `rpm`、换热器 `temp`、阀 `opening`、容器 `level`）③ 容器/塔的 `level` 液面填充 ④ `column.watches[]` 沿塔身多点读数 ⑤ `terminal` 料流箭头方向 ⑥ `props`（instrument 的 `tag`/`display`、dialgauge 的 `face`、vessel 的 `agitator`）。**源图某元件显示了什么（名字/数值/液位），就把对应字段配上**，否则画面是一堆空壳。
   - **开关量绑定 → 图元两态各有不同视觉（取值映射设计）**：给元件绑布尔状态（`running`/`open`/`closed`）后，配置面板的「取值映射」把设备值判定成**两个状态**，每个状态把图元渲染成**不同外观**（泵/阀/风机/电机=运行深填充·醒目 vs 停止浅填充；阀=开/关）——面板左侧那两个图标就是该状态在画布上的**真实渲染缩略图**，不是装饰。这是核心交互：值→状态→不同图元外观。⚠️ **prop 门控的状态视觉必须配齐对应 `props`**：`vessel`（立式容器）的「运行」=**顶置搅拌电机箱深填充**，而电机箱只有 `props.agitator:true` 时才画——给 vessel 绑了 `running` 却**不配** `props.agitator:true`，运行态在**画布和配置预览里都不显示**（运行/停止两态长得一模一样，绑定视觉上等于死掉）。规则：**给 vessel 绑 `running` ⇒ 必配 `props.agitator:true`**；推而广之，任何「靠某个 prop 部件才表达状态」的元件，绑那个状态就要把对应 prop 配上。
   - **无空元件铁律（源图有值的位置必须能显值）**：源图元件标了数值（流量/温度/压力/液位/转速/开度）→ 必须在该 node 的 `bindings` 配上对应字段 **且** 列进 `inline`（塔盘多点用 `watches`、贴管裸数字用 `readout`），数值才会渲染。只摆 `type`、不配 binding/inline = 运行时永远空白框。⚠️ **数据值本身由 MQTT/mock 喂（不归你管），但「配哪些字段 + inline 列哪些」是你的渲染责任**——漏配 = 那个位置永远不显值（这正是"仪表都是空的"的根因：不是没数据，是没配 inline 字段去显示）。
2. **id 用位号**（P-201、FV-201，全图唯一，重复会被 zod 拒），`label` 放中文设备名（用户数据，不进 i18n）
3. **布局**（世界坐标，原点左上 y 向下，x/y 是图元**中心**；视图自动 fit，只管相对位置）：
   - 主物流左→右，设备水平间距 130–150px
   - 重力流/罐底出料垂直向下（罐壳高约 100px，下游放 y+140 左右）
   - 仪表放被测设备斜上/斜旁约 70–120px，用 `lead:true` 细虚线信号线连回设备
   - ⚠️ **别让两条管在近同一高度并行贴近**（y 相差 ≲ 一个本体高 ~30px）：auto-route 会把它们叠成一条「双线」假象，高倍下像渲染 bug。实战踩过：一条「液位阀→缓冲罐」的长横管恰好与「再沸冷凝→V-106」的横管只差 5px 同高、还穿过中间设备本体 → 看着是双线。**修法**：把成簇设备分到不同 y 行，或把被某条长横管穿过/贴近的设备整体上/下移开那条线的高度；**或让其中一条线先垂直离开节点再转向**（如侧线采出从塔出来先竖直下行、再去冷却器），与另一条水平段错开高度，平行即消。用 `$hmi-visual-selfcheck` 的 **`PARALLEL` 检测**（同向段间距 4–30px + 投影重叠 >40px）专抓这种双线——比肉眼可靠（共线 `COLLIDE` 检测抓不到并排的）
   - ⚠️ **阀/终端顺序要顺流向，防「倒流自叠」**：一串 `源→阀→出口` 的元件，坐标要**单调同向**。若把阀放在源的上游侧、而出口在下游侧（如 `V-104` 在阀**右**、`VENT` 又在**更右**），管线会「先左拐到阀、再右拐去出口」原路折回、自己叠自己（一段双线）。按真实流向排坐标——`V-104 → PV(阀) → VENT` 一路同向（阀在 V-104 正上、VENT 在阀右），别让中间阀逆位
   - ⚠️ **管线/引线零重合（流向清晰铁律）**：任意两条线（管-管、引线-管）不得共线重叠或紧贴并行。重合 = 实线盖虚线（引线消失）或两管叠一条，流向不可读。布局时给每条线留独立路径；引线要从目标**没有工艺管的那一侧**垂直插入（目标某侧已被管占用就把仪表摆到空闲侧）。**落图后必须用脚本检测**：`renderScene` 返回每条边的真实折线 `edgePaths`，逐对检查 lead 段与 pipe 段是否共线重叠（同 x 竖直段且 y 范围相交、或同 y 水平段且 x 范围相交）；有重合就挪相关仪表/设备直到零重合。
   - ⚠️ **元件零重叠（铁律，与零重合并列）**：任意两个节点的**本体框 + 顶部位号标签**都不得相交（`readout`/`instrument` 气泡、阀、容器及它们的 label 都算）。密集簇（塔顶冷凝、塔釜再沸）务必拉开——节点中心间距 ≥110px、标签互不压。**落图后用脚本检测**（见下「布局自检脚本」），`OVERLAP` 清零是硬指标，比肉眼靠谱。用户反复强调「元件之间不要重叠」，这条不可妥协。
   - ⚠️ **管线/引线不穿无关元件本体（构图清晰铁律）**：除了与某元件**连接的端点**，任何管线/引线都不得从**无关元件的本体框中间穿过**（线压着设备过去 = 看不清谁连谁、错综复杂）。检测：`edgePaths` 每段若两端都不属于某节点、却落在该节点本体框内 → 判穿越，挪线或挪元件。目标是**用户一眼看清的图、不是缠成一团的图**。⚠️ **`CROSS` 必须真清零，不许借口「可接受」放行**：引线穿设备**永远能靠摆位消除**（见下控制器摆位法则）。把它当「擦个框角无所谓」是错的——离线示意图线画得细，看着像擦角；**真 Canvas 上是一条线劈开设备本体，极丑**。只有极密工厂的**管-管**交叉才可能真无法避免、那时才「朝最少交叉优化」；**引线穿设备（CROSS）没有这个豁免，一律挪到 0**。任何挪动不得偏离原图拓扑与相对位置（还原场景构图服从原图，不为美观改结构）。
   - 📍 **控制器摆位法则（根治引线穿设备）**：控制器要同时连「阀」和「被测设备」两根线。把控制器放在**「阀-设备对」的正下方（或外侧）居中**，让两根引线**朝相反方向分叉**——一根斜上到设备、一根斜上到阀；**别把控制器塞在阀的正上方或正对面**，那样去对面目标的引线必穿过阀/设备（`CROSS`）。实测：`LIC` 放在 `E-115A`(左) 与 `LV_113A`(右) 这一对的**正下方**，测量线左上进换热器、控制线右上进阀，零穿越；而控制器一旦在阀正上方，测量线去对面换热器就必穿阀。
   - ⚠️ **枢纽设备多管用 `fromSide`/`toSide` 分散**：塔/大容器常 6+ 条管接入，auto-route 默认全收口到**节点中心** → 在中心高度叠成一束「假双线」。给每条管显式标 `fromSide`/`toSide`（值 `"L"|"R"|"T"|"B"`）分到不同边：进料 `toSide:"L"`、塔顶气相 `fromSide:"T"`、侧线采出 `fromSide:"R"`、再沸/塔釜采出 `fromSide:"B"`、回流返塔 `toSide:"T"`。注意 **同一边仍会让多管共壁一段**（`sideRoute` 收口该边中点，不按目标高度分布），所以一条边最多放 1–2 条主管，其余改走相邻边或用自由点。**落在节点本体框内的共线段（如多管共用塔中线 x=塔心、被塔体盖住）不算视觉重合**，检测时按"段是否落在某节点框内"过滤掉。
   - 📐 **布局自检脚本（免起浏览器，强烈推荐）**：`buildScene`→`renderScene` 取节点框（`registry.get(type).bounds(node)` 再按 `sizeX/sizeY` 围绕中心缩放）+ `edgePaths`，脚本自动报 `OVERLAP`（节点框两两相交）与 `COLLIDE`（同向段共线重叠），再 `rsvg-convert` SVG→PNG 肉眼复核。完整脚本见 `$hmi-visual-selfcheck`「离线渲染自检」。比连 DB 起 `dev:preview` 快、且不污染默认图。
4. **连线**：每条连线**必须给 `from`/`to` = 两端节点 id**（必须是 `nodes` 里真实存在的 id）。
   - ⚠️ **auto-route 靠 `from`/`to` 每帧按设备实时位置重算并自动收口**；`points`（≥2 点折线）只是断链兜底快照，不是连接依据。
   - ❌ **只给 `points` 不给 `from`/`to`，或 `from`/`to` 写了不存在的 id** → 退回静态 `points`，线**不跟随设备、对不上、看着没连**（这正是"有的线没连起来"的常见成因）。
   - ✅ 正确：`{ "id": "e1", "from": "P-201", "to": "V-201", "points": [[x1,y1],[x2,y2]] }`——`from`/`to` 都能在 `nodes` 里找到。
   - `lead` 边（仪表引线，细虚线）：`from`=仪表节点、`to`=被测设备节点；显示为虚线属**设计如此**，不是断线。
   - `flowBy` 可配但 `SHOW_FLOW=false` 全局关闭——配了属语义占位，别当验收项。
   - 端点 `points` 落到设备边缘附近即可（渲染按设备实时位置重算并收口，精确数值不重要；真相是 `from`/`to`）。

### 连线总则：先判「这是哪种线」，再抄对应模板（弱模型逐条机械照做，别凭感觉）

> 这一节是连线最易错的部分，**每画一条线都按下面 4 步走，不要跳**：
> **第 1 步 判线种** → **第 2 步 抄模板填字段** → **第 3 步 选边/选点** → **第 4 步 落图后脚本检测**（见本节末）。

**第 1 步：每条线只可能是这三种之一，先对号入座**

| # | 线种 | 何时用（判据，二选一就够） | schema 长相 | 端点落在哪 |
|---|---|---|---|---|
| ① | **实线管道** | 走**工艺介质**的管（料/汽/水/冷媒/产品…）。两端是设备/阀/泵/容器/`terminal` | `lead` 不填 | 两端**节点**，用 `fromSide`/`toSide` 指定边 |
| ② | **元件引线** | **气泡仪表**关联某设备/阀/容器的**标准接口**（测它/控它） | `lead:true` | 两端**节点**，`fromSide`/`toSide` 指定边（四边都行） |
| ③ | **自由标注线** | 虚线要落到**元件本体的「任意位置」**（非标准接口）——典型：塔身某塔板高度、管段中间某点 | `lead:true` + 自由点 | `fromPoint`/`toPoint` 写**世界坐标**，不连节点 |

**② vs ③ 只问一句**：这条仪表虚线指向的是元件的**标准接管口（一条明确的边）**→ ②；还是元件**本体上某个具体位置/高度（塔的第 N 板、管中段某点）**→ ③。**原图里塔旁一排仪表各指一个塔板高度 = ③**（用②连塔节点会全收口到塔边中点、挤成一束 = 错）。

**第 2 步：抄模板填字段**（`points` 一律占位，渲染按实时位置重算；自由线的 `points` 要等于两个自由点）

**① 实线管道**
```jsonc
{ "id":"e1", "from":"FEED",    "to":"FV_012A", "toSide":"L",                 "points":[[0,0],[0,0]] }
{ "id":"e2", "from":"FV_012A", "to":"T-104",   "fromSide":"R", "toSide":"L", "points":[[0,0],[0,0]] }
```
铁律：
- **泵/阀只有左右口**：接泵/阀的实线 **必须** `fromSide`/`toSide`=`"L"` 或 `"R"`，**绝不 `"T"`/`"B"`**（阀从顶接管=不符实物）。料一侧进、另一侧出。
- **枢纽设备（塔/大罐）多管分边**：一条边最多放 1–2 根主管，其余分到别的边或用自由点——否则多管在该边中点共壁成「假双线」。例：进料 `toSide:"L"`、塔顶汽 `fromSide:"T"`、侧采 `fromSide:"R"`、塔釜采出 `fromSide:"B"`、回流返塔 `toSide:"T"`。
- **每条管必有源有汇（不许凭空起线）**：实线链必须**起于一个源、终于一个汇**（源/汇=`terminal` 或设备）。不许「入口悬空、凭空从一个阀/换热器冒出来」——阀控的是某个进/出口的流量，那个口必须真实存在，缺就补 `terminal`（先例：`FV_019A` 再沸器热媒阀曾只有出口、入口悬空，补 `蒸汽` terminal 才成立）。落图后逐条核每条实线两端是否都落在真实源/汇上。
- **顺流向单调排坐标**：`源→阀→汇` 的 x（或 y）同向递增/递减，否则线原路折回自叠（见「倒流自叠」）。

**② 元件引线**（仪表连设备/阀的标准接口）
```jsonc
// 指示仪表 PI：一根线落被测设备
{ "id":"l-pi1",  "from":"PI_035A",  "to":"E-118",   "lead":true, "fromSide":"B", "toSide":"T", "points":[[0,0],[0,0]] }
// 控制器 FIC 的控制线：→ 它控的阀（另有测量线，见「控制回路」）
{ "id":"l-fic1c","from":"FIC_018A", "to":"FV_018A", "lead":true, "fromSide":"R", "toSide":"T", "points":[[0,0],[0,0]] }
```
铁律：
- **必写 `fromSide` + `toSide`**（仪表侧 + 设备侧）。⚠️ **不写 = 渲染每帧按相对位置自动选边 → 用户拖仪表时连接边会翻（顶↔侧跳变）= 违反端点钉死 = 错**。批量生成最容易漏这俩字段，逐条核。
- 引线**不受 L/R 约束**，设备端 T/B/L/R 四边都行；挑「朝向仪表、且没被工艺管占用」的那条，末段垂直插本体最干净。
- 仪表斜放也行，但**同行（正旁边=水平）/同列（正上下=竖直）摆位最干净**（端口对齐→直线无折）。

**③ 自由标注线**（连塔身等本体任意位置）
```jsonc
// PI_035A 连塔身「它自己的高度」：两端都是自由点，水平落到塔左壁
{ "id":"l-pi35", "lead":true,
  "fromPoint":[460,170],     // 仪表气泡朝塔那侧的边
  "toPoint":[610.77,170],    // [塔壁x, 仪表自身y]：x=塔壁、y=仪表 → 水平线落塔身该高度
  "points":[[460,170],[610.77,170]] }
```
铁律：
- **绝不写 `from:塔节点`/`to:塔节点`**（会收口到塔边中点，多线汇聚成一束）。
- 塔端 `toPoint=[塔壁x, 仪表自身y]`：塔壁 x 取 `column.coreBox` 的左/右缘（朝向仪表那侧），y 取该仪表的 y → 每条线水平、沿壁按各自高度分布。
- 自由点**必须精确压在目标几何上**（塔壁 x 准确、或管线真实折线上），差几像素就空隙——取真坐标见「零空隙铁律」。
- 工具：`scripts/free-column-leads.mts` 把存量「连塔节点」的引线批量转成自由线。

---

**第 3 步要点：端点钉死 + 怎么折**

**端点钉死铁律（①②③ 通用，最高优先）**：从元件连接点拉出的线（①②），端点**钉在边中点不动**——拖一端，另一端连接点**原地不动**，只重算线形。**没有「浮动对齐 / 沿元件边滑动取点」**（曾做过 floatPortOf 浮动对齐，拖对端时点沿边滑走，已废：`edge-route.ts sideRoute` 一律 `portOf` 固定中点）。③ 自由线端点 = 你写的坐标，也钉死在那坐标。**想让线落「本体任意位置」≠ 让元件连接点浮动**（错方向，别加 multiPort 之类）——用 ③。

**怎么折（端口定了，线形自动算，照这个规律预判即可）**：
- 一端水平口（`L`/`R`）+ 一端竖直口（`T`/`B`）→ **单折 L**，拐角=`[竖直端.x, 水平端.y]`（竖直口的竖直线 × 水平口的水平线 的交点）。
- 两端同水平（`L`↔`R`）或同竖直（`T`↔`B`）→ **中点 Z 两折**。
- 两端口对齐（同 x 或同 y）→ **直线无折**（所以同行/同列摆位最干净）。
```
单折 L 例：A 从 T 口(顶)出、B 从 L 口(左)入，B 在 A 右上方
   A.T(20,0)
      │            拐角=[A.x, B.y]=[20,80]
      └──────────● B.L(200,80)      ← 竖直段 x=20、水平段 y=80
```

**第 4 步：落图后必做连线检测** —— 见本文档「管线/引线零重合」「不穿无关元件本体」两条铁律里的 `renderScene` 脚本检测法（逐对查 `edgePaths` 是否共线重叠 / 落在非端点节点框内）。改完再 `npm run dev:preview` 截图亲眼复核。

**连线完整小例（覆盖 ①②③ 三种线，照搬这个结构）**

布局：`FEED`(terminal,100,300) — `FV_1`(controlvalve,250,300) — `T_104`(column,450,300，左塔壁 x≈430)；`LIC_1`(instrument,250,430，阀正下方)；`PI_1`(instrument,320,180，塔左上)。

```jsonc
"edges": [
  // ① 实线管道：料 FEED → 阀左进、阀右出 → 塔左（阀只有 L/R 口）
  { "id":"e1", "from":"FEED", "to":"FV_1",  "toSide":"L",                 "points":[[0,0],[0,0]] },
  { "id":"e2", "from":"FV_1", "to":"T_104", "fromSide":"R", "toSide":"L", "points":[[0,0],[0,0]] },
  // ② 元件引线：LIC 控制器两根——控制线↑到阀、测量线→到塔釜液位；都连节点、都写双 side
  { "id":"l-lic-c", "from":"LIC_1", "to":"FV_1",  "lead":true, "fromSide":"T", "toSide":"B", "points":[[0,0],[0,0]] },
  { "id":"l-lic-m", "from":"LIC_1", "to":"T_104", "lead":true, "fromSide":"R", "toSide":"B", "points":[[0,0],[0,0]] },
  // ③ 自由标注线：PI_1 测塔身「它自己的高度 y=180」(某塔板) → 两端写坐标、落塔左壁、不连塔节点
  { "id":"l-pi", "lead":true, "fromPoint":[350,180], "toPoint":[430,180], "points":[[350,180],[430,180]] }
]
```
- e1/e2 走 `L`/`R`（阀只有左右口）；l-lic-* 两根引线连节点且**各写 `fromSide`/`toSide`**（不写会翻边）；l-pi 是自由线（连塔身任意高度，两端写坐标，`points` 等于那两点）。
- 控制器两根线 `fromSide` 不同（控制线 `T`、测量线 `R`）→ 自然 L 形分叉、不叠成一条（见「控制回路」）。

### 仪表（instrument）连线：连什么 = 由仪表功能决定（语义铁律）

**每根 `lead` 引线的连接对象只有两类：① 元件节点（设备 / 阀 / 容器；**另一台仪表仅限串级、少见，默认不连**）② 管线上的点（自由端点 `fromPoint`/`toPoint`）。** 连哪个**由这台仪表测什么、控什么决定**——不是凭就近乱连。连错 = 工艺语义错：用户拿这图改时靠连线判断「这台仪表测哪儿、控哪个阀」，连错了根本读不懂、改起来累死。

| 仪表测/控的量 | 测量线落在 | 典型错误（禁） |
|---|---|---|
| 流量 `F*`（FIC/FR/FT） | 被测**管段**上的点（自由点压管；流量控制取阀的**上游进口段**） | ❌ 落到阀/泵本体——流量是管里介质的量，不是阀/泵的属性 |
| 液位 `L*`（LIC/LT） | 被测**容器**元件（塔 / 罐 / 缓冲罐） | ❌ 落到管线 |
| 温度 `T*` / 压力 `P*`（TIC/PIC/PI/PDI/TI） | 被测**设备**元件 或 **管段**点 | ❌ 落到无关元件 |
| 质量 / 分析 `AI` | 取样的**产品 / 采出管段**点 | ❌ 落到泵 / 储罐本体 |

- **纯指示仪表**（不带「C」：PI/TI/FI/LI/AI）= **一根** `lead`，落**被测物**（按上表）。指示仪表**没有控制线、绝不连阀**（流量指示 `FR` 落进料管、不是连进料阀；质量 `AI` 落采出管、不是连泵）。
- **控制器**（带「C」：FIC/LIC/TIC/PIC）= **两根** `lead`：控制线→**阀**（元件）、测量线→**被测物**（按上表）。细则见下「两根线铁律」。
- **串级（cascade）是例外、默认不画仪表→仪表**：⚠️ 气泡仪表**通常连阀、管、设备/容器，基本不连另一台气泡仪表**——两个仪表框之间拉一根长线在画面上很怪、也不符常规读图习惯。**默认让每个控制器各自连自己的阀 + 被测物（管/设备/容器），不要互相连**。仅当工艺确有串级、且没有更直接画法时，才破例用串联（主控制器引一根线到副控制器，技术上合法但少用）。例：塔釜液位**优先直接控塔釜采出阀**（`LIC`→阀 + 测塔釜容器），而不是 `LIC`→`FIC` 串级；容器液位也常由容器自身的 `level` 填充直接显示、无需额外控制器连线。
- **落点用「功能反推」定，别凭位置就近连**：拿到图先对每台 instrument 问两句——「它测什么量？」（→查上表定测量线落点）「它控哪个阀？」（→控制线落那个阀）。位置就近乱连是连线无规则的根因。

> **✅ 连线四问（每台 instrument 落图前过一遍，全过才算对）**：①带「C」吗？→带C=2根(控制线+测量线)、不带C=1根。②测量线落对物了吗？→流量落管、液位落容器、温压落设备、质量落采出管；**指示仪表不连阀、流量不连泵**。③控制线落到「它的」阀了吗？（仅控制器）④误连到另一台仪表了吗？→默认不该有（串级才例外）。**摆位**：控制器放「阀-设备对」正下方、两线分叉斜上、`CROSS=0`；最后**真 Canvas 截图核一眼**（离线图会把线穿设备显成擦框角）。

### 控制回路连线：仪表↔阀的两根线（铁律）

**控制器仪表（FIC/LIC/TIC/PIC 等带「C」的）连接它控制的阀时，画面上必须是两根 `lead` 边，不能只画一根。**
原因：一台控制器同时承担两件事——操纵阀 + 测量被控量，两件事各一根线。

| 线 | `from` | `to` | 语义 |
|----|--------|------|------|
| 控制/信号线 | 仪表节点（FIC_xxx） | 对应阀节点（FV_xxx） | 仪表输出信号去操纵阀 |
| 测量线 | 仪表节点（FIC_xxx） | 被测点（自由端点压在被控管段上） | 仪表读取被控变量 |

**两根线呈 L 分叉、不可同向（关键摆位，否则两线叠成一条）**：把控制器摆在**阀与被测点之间的夹角**位置，使控制线、测量线一横一竖（或一左一右）岔开。别让两根都朝同一方向——`renderScene` 检测会报这两条 `lead` 的 `COLLIDE`。实测：FIC 摆在阀的一侧、被测进口管在另一方向时，控制线走一轴、测量线走另一轴，自动 L 形分叉；若发现两根 lead 同向贴着走，把仪表挪到「阀」与「被测点」连线的拐角处即可分开。

**测量线落点 = 语义精确（错一侧 = 还原错误）**：

- **流量控制（FIC/FR）**：测量线落在**被控管段的进口侧**（阀的**上游**那段管，料流进入阀的一侧），**绝不落出口侧**。进口 vs 出口是两个物理量；落错侧 = 测成了出口流量 = 还原错误。「左边管道 ≠ 右边管道」，要看料流方向。
- **液位控制（LIC）**：测量线 → 被测**容器节点**；控制线 → 控制的阀节点（LV_xxx）。
- **压力/温度（PIC/TIC）**：测量线 → 被测点（塔/换热器/管段上的自由点）；控制线 → 阀节点。
- **纯指示仪表（PI/TI/PDI/FI 等，不带「C」，不控阀）= 一根 `lead` 线**落被测物，不存在第二根。

### 自由端点必须精确落在管线真实几何上（零空隙铁律）

所有连接端点必须**零空隙**落在目标上：

- **节点端点**（`from`/`to` 写节点 id）：auto-route 自动收口到节点边缘，无需手工对齐。
- **自由端点**（`fromPoint`/`toPoint` 写坐标）：必须**精确压在该管线的真实折线上**——浮在管线附近（坐标没压在线段上）会导致引线端点悬空、出现肉眼可见的空隙（lead 离管线差几像素）。

**如何取到精确坐标（勿凭感觉猜）**：
交互画线：把端点拖到管线上，系统自动 pipe-snap 吸附到该管线精确点，零缝。
手写 schema：管线的真实渲染折线由两端节点的 auto-route 决定，不是想当然的直线。**用临时 `console.log` 在 `scene-render.ts` 边循环中打印目标 edge 的 `points` 数组取真坐标，再把自由点定位到该折线上**（验完删 log，`grep console.log` 确认无残留）。靠"估算坐标"几乎一定有空隙。

     - **演示模式在 app 里只读、不可编辑**（生产无 `VITE_DEMO_EDIT`，演示态禁编辑）；**DB 里有真实图时，顶栏「演示」切换自动隐藏**（`demoAvailable={demoScene}`），只剩「编辑/预览」显真实数据。所以「演示」永远是那张固定样板，既不会被客户图污染、也不会反向同步进编辑/预览。
     - ⚠️ 踩过的坑（实拍）：平台 AI 直接拿演示样板二次修改、还同步进了编辑/预览——**错**。记牢：demo=固定样板（仅参考），default-mimic/DB=客户真实图（生成目标），两者井水不犯河水。
   - **`default-mimic.json` 现在只是 DB 空表时的空白种子**（开局空画布），**不再是客户图的生成目标**——别往里写客户 schema。客户图一律用上面的 `load-mimic.mts` 直接 UPSERT 进 DB（不靠 seed，DB 非空也能覆盖，无需清表）。`example-mimic.json` 与 `HMI_SEED_DEMO` 已移除。

   - `broker` 字段已被运行时忽略（连接走环境变量，见 $hmi-data-binding），schema 层仍兼容但新建 schema 不要填写

## 照图忠实还原（image → 1:1 复刻）

客户给参考图要"还原"时，目标是**位置坐标 + 朝向 + 元件类型 + 图上文字都 1:1 对齐源图**，不是凭印象重画。实战要点：

> **先分清两种场景**：① **照图 1:1 还原**（客户给了指定工艺图）→ 逐元件忠实，本节规则全适用。② **做演示/样板图**（无指定客户图、自拟示例）→ 在忠实基础上**可适度精简**：砍冗余对称组（双列对称设备留一组示意）、次要辅助小回路、纯重复读数，聚焦主工艺流。目标画面**饱满又不拥挤**——既不堆满几十件难辨识、也不稀疏到大片空白（实测 75 件嫌乱、27 件嫌空，~40 件刚好）。精简后仍保留**代表性图元类型**（塔/换热器/罐/泵/调节阀/变送器/终端各留代表）展示模板能力；留下的元件**位号/文字照样逐字真实**——精简是减数量，不是放宽逐字照搬。

- **🔒 图上文字逐字照搬，绝不改写（头号铁律，优先于一切）**：源图上用户写的每一处文字——设备名（写「设备A」就是「设备A」）、位号、介质名、工段/料流标注——必须**一字不差**搬到对应元件（`node.label` / `props.tag` / node `id`）。**严禁**改名、改写、意译、翻译、补全缩写、"优化润色"、或自行杜撰一个"更专业"的名字：用户写「设备A」**绝不能**变成「反应釜」「进料泵」之类。**还原 = 忠实复刻，不是自由发挥**——用户正是靠这些原文字辨认哪台是哪台，你一改他就对不上自己画的图、压根不知道哪是哪。某处文字看不清/不确定时，**标注存疑问用户**，宁可留空也别瞎填一个。这条凌驾于"显示美观/命名规范"等一切之上。

  **元件层面的逐字照搬规则（具体落法）**：
  - **方框仪表（`props.display:"box"`）**：方框渲染 = **位号（顶部深色带 = `node.id`）+ 实时值**，下方 `node.label` 留空或不填——因为源图方框就只有位号+值，**没有中文名**。给 `node.label` 安中文名 = 杜撰，违规。仅当源图方框下方**本身就写了中文名**时才填。
  - **阀/设备/容器**：`node.label` = 图上的**位号/标签**（如 `FV_022A`/`E-115A`/`V-106`），不是中文描述。位号即 `node.id`，`label` 可与 `id` 相同或保持与图一致。
  - **终端（`terminal`）**：`node.label` = 图上的文字（`FEED`/`Venting`/`VACUUM PUMP`/`REF`/`RES`/`CWR`/`S3S`/`SC3`/`TK-001`…），一字不差，大小写照搬。
  - **仅当源图本身就写了中文名**时才填中文名；图上没有就不许造——不存在「意思对了就行」的豁免。

- **先逐区精准识别再落点**：源图每个符号对应哪个 `type`（卧罐 vs 立罐、调节阀 vs 手阀、变送器 vs 表盘），错认一个就"不像"。拿不准对 `capabilities.ts` 核。
- **直接用源图像素坐标当世界坐标**：视图自动 fit，照着图比例摆即可，别自己另起一套布局。
- **塔/高瘦设备只能等比缩放（`sizeX = sizeY`，铁律）**：`column` 造型自带瘦高比 H/W≈3.75，要放大到贯穿画面就**等比**放大（如 `sizeX:3.6, sizeY:3.6`），瘦高靠造型本身的比例。⚠️ **绝不可用非等比 `sizeX≠sizeY` 去「拉瘦/拉高」**——那会把塔盘、上下封头、侧面管嘴全按 y/x 不同倍率压扁，叠加造型自带的 3.75 后视觉比例可达 ~10:1，塔严重变形失真（这是实测踩过的坑：`sizeX:1.25, sizeY:3.6` 把塔拉成变形细条）。画布手柄拖拽本就锁等比，schema 手写也必须守等比。塔不够细高是造型上限，不能靠拉伸硬凑。
- **塔盘温度贴塔身（多点、不同高度，别精简删光）**：精馏塔沿塔身**不同高度**有多个温度测点（进顶/进顶1/进中上/塔中部/塔中下/塔釜…），是塔的关键工况显示，**必须齐全还原**。两种画法：① `column.watches[]`（带 `unit:"°C"`）沿塔身竖直均布；② 要精确单点位置用独立 `readout` 沿塔身贴放，**位号照点位表真值**（`TI_059A`/`TI_061A`/`TI_062A`/`TI_063A`/`TI_064A`…，不用自编序号）。⚠️ 演示精简时可砍辅助回路，但**塔身温度点不能删到只剩一两个**——塔上没有分布的温度数值点，就不像在监控精馏塔了。
- **仪表引线一律 `lead:true`**（细虚线信号线），且**必须显式写 `fromSide`/`toSide` 把两端钉死**（仪表侧 + 设备侧，值 `"L"|"R"|"T"|"B"`）。⚠️ **不写 side = 渲染每帧按相对位置 auto 选边 → 用户拖动仪表时连接边会翻（顶↔侧跳变）= 违反「端点钉死」铁律**（实线管道两端也务必写 side，同理）。**选边规则**：仪表在设备**上/下**→仪表 `B`/`T`、设备 `T`/`B`（竖直对插）；仪表在设备**左/右**→仪表 `R`/`L`、设备 `L`/`R`（水平对插）；斜放则一端竖一端横 = 单折 L（拐角见连接点规则的折痕公式）。`from`=仪表、`to`=设备；`points` 占位无所谓（渲染按实时位置重算，但**连接边由 side 钉死、不随拖动变**）。**同行（正旁边）/同列（正上下）摆法最干净**（端口对齐→直线无折）。画布拉线工具连 `instrument` 会自动 `lead` 并存 side（交互产出本就钉死；唯独**手写/批量生成的 JSON 易漏 side**，务必补全）。
- **保留源图重要文字（位号/点位号）**：源图标在仪表上的**完整位号**（如 `PI-035A`、`LIC-113A`）是关键信息，必须搬上画布——别只用 ISA 简写（`PI`/`LIC`）当 `props.tag`，那样满屏 `LIC`/`FIC` 撞车分不清是哪台。把真位号写进 node `id`，并视画面风格选显示方式（下条）。同理源图的关键工艺标注（介质名、设备号）也尽量还原。⚠️ **多点位也必须用真实位号、严禁自编序号**：塔身多塔盘温度、多组对称设备等，逐一对应点位表真实位号（如塔身温度逐层 `TI_059A`/`TI_061A`/`TI_062A`/`TI_063A`/`TI_064A`），**绝不可**用 `RT-1`/`RT-2`、`T1`/`T2` 等自编序号充数——序号是杜撰（违反逐字照搬），运行时还对不上真实测点 topic（值永不更新）。点位表（如 SUPCON 的位号说明表）就是真值来源，照搬。
- **DCS 方框风格的仪表**：源图若是 SUPCON 等 DCS 画面、仪表是「位号 + 数值」方框（非 ISA 圆圈），给 `instrument` 设 `props.display:"box"` → 渲染成 DCS 数据框：顶部深色带显**完整位号(node id)**、下方显实时值（越限变色）、中文名留框下方。默认（不设）仍是 ISA 圆气泡（圆内放 `props.tag` 简写）。一张图统一用一种风格。
- **管线没有颜色**：所有工艺管线统一描边色，schema 无 per-edge 颜色字段——源图的红/蓝/橙管线**还原不了**，别硬凑，照实说明。
- **重叠是"难看"主因**：仪表气泡+位号标签占位不小，密集区（塔顶冷凝/缓冲罐簇）务必拉开 ≥80px，落图后亲眼查重叠再调。
- **照图还原文字（填进元件自身，不另起标注组件）**：识别源图标注的文字信息，**填进对应元件本身**让其呈现，不要为此新建独立标注组件。规则：① `instrument` 默认渲染为 DCS 方框（顶部深色带显完整位号=`node.id` 或 `props.tag`、下方显实时值越限变色、中文名=`node.label` 在框下方），源图若是 ISA 圆气泡风格则设 `props.display:"box"` 与之对齐（默认已是 box；`display:"bubble"` 回退圆圈）；设备/阀/容器等显示 `node.label`（中文名或位号）+ 内联值 + 绑定值。② **实时数值**（流量/温度/压力/液位/阀位 %）→ 绑 topic（`instrument` 值、设备内联值；自由摆放的裸数值用 `readout` 并绑一个值）；③ **静态名称/位号**（介质名、工段名「至/来自 XX」、设备别名）→ 填 `node.label` 或 `props.tag`（不绑数据）。重要文字全部搬上画布；绝不把静态名称乱绑 topic、也不把实时值做成死 label。

- **原图文字逐类还原（先盘点全图文字、再逐类落机制；只用现有组件，不加纯文字元素）**：工业图上文字很多，漏还原=还原不完整。先把源图所有文字圈一遍列清单，逐条对到下表机制：

  | 源图文字 | 是什么 | 还原机制 |
  |---|---|---|
  | 设备位号（`P-107A`/`E-115A`/`V-104`）| 设备标签 | `node.id`（+ `label`），方框仪表顶部带自动显 id |
  | 仪表位号（`FIC_018A`/`LIC_113A`）| 仪表标签 | `instrument` 的 `node.id`（DCS 框顶部深色带）|
  | 实时数值（流量/温度/压力/液位/开度）| 测点值 | `bindings`+`inline`（设备内联）/ `instrument` 值 / `readout`（贴管裸数字）/ `column.watches`（塔盘多点）|
  | 介质/物料名（蒸汽/`CWR`/`SC3`/`S3S`）| 管线介质标注 | 边界处 `terminal.label`，或就近设备 `node.label` |
  | 工段/去向（`FEED`/`Venting`/至XX工段/`Product`）| 边界 I/O 文字 | `terminal.label`（`rotation` 定朝向）|
  | 塔各塔盘温度（多个℃读数贴塔身）| 多点测温 | `column.watches[]` 沿塔身均布，或多个 `readout` |
  | 设备中文名/别名 | 源图写了才有 | `node.label`（源图没写就不填，守逐字照搬）|
  | 区段标题（`TOWER`）/图例/颜色说明 | 画面 chrome，非工艺 | **跳过**（不是工艺信息；现无纯文字元素，硬塞 terminal 反而乱），交付说明里讲明跳了哪些 |

  ⚠️ 别对成片文字视而不见——**圈不到机制的只有纯装饰**（标题/图例），其余介质名、工段名、位号、多点数值都有现成机制承载，必须还原。
- **自由端点管线（`fromPoint`/`toPoint`）**：连线端点既可以是节点 id，也可以是画布上的自由世界坐标。还原「阀控制某段管道流量」等细节时，把控制/测量**虚线（`lead:true`）落在被控管段的具体点**（如阀**左侧进口管段**），而非阀中心节点；实线管道同理可连到任意位置（off-page 工段桩、管段中点等）。自由端不跟随任何节点移动，适合钉住管线上的固定点。
- **进口 vs 出口 = 两层对齐**：几何层——虚线引线落进口管段（用自由端点钉位，不是出口管）；数值层——FIC 仪表的 `value` 绑**进口流量计** topic（绑定 `path` 已支持）。两层都要对：只画对线却绑了出口值，或画错管段但绑对值，都是半错还原。校验时对照源图逐条核「几何落点 + 绑定 topic」。

## 坑速查

| 坑 | 真相 |
|----|------|
| 客户图建在演示样板上改、还同步进编辑/预览 | **错**：demo=钉死只读参考(只可参考画法)；客户图**从零新生成**、跑 `load-mimic.mts` 进 DB；DB 有数据自动隐藏演示。两条线分开互不污染 |
| 文档类型表当真相源 | `capabilities.ts` 才是；文档表会滞后 |
| 给阀/泵写当前开关状态 | 拓扑不含运行态；状态=绑定步骤的事 |
| `fault` 当 binding key 绑 | `fault` 是派生态：数值字段越 `alarms` hi/hihi/lo/lolo 限自动点红环，无需也不能直接绑（详见 $hmi-data-binding） |
| 想改 DB 里已有的客户图 | 用 `load-mimic.mts` UPSERT（不靠 seed，DB 非空也覆盖 default 行）；别再编辑 json 文件等 seed |
| 指望 flowBy 动画验收 | `SHOW_FLOW=false` 全局关闭 |
| 上传/校验只看 zod 通过 | 还要跑 `validateMimicAssets` 逐条裁决告警 |
| 精确凑 edge points | 渲染会按设备实时位置重算，points 合理即可——真相是 `from`/`to` |
